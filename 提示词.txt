作为一名资深爬虫架构师，请基于 Python 3.12 对以下分布式爬虫框架设计进行架构评审，绘制系统架构图，并针对每个模块提出具体的优化建议和详细说明。
框架整体数据流：种子注入 → Redis 去重与分发 → Scrapy分布式抓取（代理池支撑） → 数据清洗校验 → 双通道写入（Kafka 实时流 + PostgreSQL 持久化，媒体文件存入 MinIO） → 日志监控与告警全链路覆盖 → API 网关 + Vue 3 前端统一管理。
设计目标：
1.使用git做代码管理工具,便于代码的更新迭代
2.支持docker-compose
3.支持热更新
4.日均千万级数据抓取
5.多机器部署统一日志，每个爬虫都有spider_id,支持全链路管理，后续需要做日志聚合检索
框架包含以下 11 个模块，请逐一评审：
1. Redis 去重及种子分发模块：去重、ZSET 优先级队列、种子 TTL 管理、Lua 原子操作（检查+添加）、批量操作减少网络往返。

2. 代理模块：多 API 源支持、HTTP/HTTPS/SOCKS5 协议区分、使用频率限制防封禁、定时刷新补充代理池。（这个模块有现成的代码后续再分析融入，暂时留个位置）

3. Scrapy 抓取模块：基于 scrapy-redis 分布式爬取、双管道输出（Kafka + PostgreSQL）。（这样设计合理？）

4. 日志模块：JSON 结构化日志、trace_id 全链路追踪、分级存储（ERROR 单独输出）、模块隔离独立日志文件、异步写入（enqueue=True）、远程日志（Kafka/Syslog）、100MB 自动轮转、30 天自动清理。

5. 数据写入模块：PostgreSQL 批量插入（自动构建 INSERT SQL）、MinIO/S3 上传（字节内容或 URL 下载）、Kafka JSON 序列化发送、内存写入缓冲、指数退避重试、Schema 自动建表、Kafka 按 spider_name/域名分区、Redis 死信队列支持重放。

7. 日志监控模块：watchdog 实时监听、增量读取、关键词匹配告警（ERROR/CRITICAL/Exception/Traceback）、钉钉/飞书 Webhook/邮件/Prometheus+Alertmanager 多通知渠道、告警聚合防风暴、告警分级、统计面板、Redis/Kafka/PG 健康检查探针。

9. API 网关模块（FastAPI）：任务 CRUD 与启停、种子队列管理与批量导入、代理池状态查询与刷新、监控大盘（成功率/错误统计/资源使用）、PostgreSQL 数据分页查询与过滤排序、健康检查、CORS 支持、Vue 3 静态文件托管、Pydantic 请求验证。
